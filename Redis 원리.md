# Redis 원리(순서는 먼저 Redis를 보고 짬)
![alt text](image-61.png)

Redis 명령 프롬프트(cmd) - get, set, ttl, flushall, keys *

![alt text](image-60.png)

이 번호 순서대로 데이터가 어떻게 이동하는지 이해하면 원리가 한눈에 보입니다. 왼쪽의 검은색 네모는 애플리케이션(서버)이라고 생각하시면 됩니다.

🔄 캐시 어사이드(Look-Aside) 작동 원리
사용자가 서버에 "데이터 좀 보여줘!" 하고 요청했을 때, 서버는 다음 4단계로 움직입니다.

1. 1번 과정 (Cache Miss 확인)
서버는 먼저 데이터가 Redis(캐시)에 있는지 확인합니다.

만약 Redis에 데이터가 없다면? 이를 '캐시 미스(Cache Miss)'라고 합니다.

2. 2번 과정 (DB 조회)
Redis에 원하는 데이터가 없으니, 서버는 어쩔 수 없이 진짜 데이터 저장소인 RDB(Oracle, MySQL 등)로 찾아가 데이터를 직접 조회합니다.

3. 3번 과정 (데이터 반환)
DB가 요청받은 데이터를 찾아서 서버(애플리케이션)에 보내줍니다.

4. 4번 과정 (캐시 업데이트)
서버는 DB에서 가져온 따끈따끈한 데이터를 Redis에 저장해 둡니다. (다음 요청 때는 DB까지 안 가고 1번에서 바로 끝내기 위해서죠!) 그리고 사용자에게 데이터를 최종 전달합니다.

💡 왜 이렇게 할까요? (이 패턴의 장점)

처음만 느리고 그 다음부턴 초고속: 첫 번째 요청(Cache Miss) 때는 DB를 거쳐야 해서 조금 시간이 걸리지만, 두 번째 요청부터는 1번 과정(Redis 조회)에서 바로 데이터를 주기 때문에 엄청나게 빨라집니다.

DB의 부담을 덜어줌: 수많은 사용자가 동시에 몰려도 DB로 가기 전에 Redis가 앞단에서 다 막아주기 때문에 DB가 뻗는 걸 방지합니다.

Redis가 죽어도 서비스는 안전: 만약 Redis 서버에 문제가 생기더라도, 애플리케이션은 2번 과정처럼 DB로 바로 직행해서 데이터를 가져올 수 있으므로 서비스가 완전히 마비되지 않습니다.

쉽게 요약하자면, "자주 찾는 책은 서가(DB) 깊숙이서 매번 꺼내오지 않고, 책상 위(Redis)에 올려두고 바로바로 꺼내본다"는 원리입니다!
